Overview

Open Source Solution

The Modeling Health Tools (MDHT) and the Open Health Tools are dedicated to providing 100% open source solutions within the Health Information Industry.  The continuing support of the Eclipse Papyrus project is result of the ongoing effort to promote and use open source solutions. 

What is the Eclipse Papyrus Project?

The Eclipse Papyrus is an open source component of the Model Development Tools (MDT) subproject to provide an integrated, user-consumable environment for editing models based on UML.  For a more complete description of the Papyrus Project please see MDT Papyrus Proposal.

The project was scheduled to be officially release as part of the eclipse Helios release in June 2010, see project plan MDT Plan.  As such, the current Papyrus application is still under development and is not been officially released.  The MDHT project has been monitoring the progress of Papyrus project and integrating at release intervals in order to test MDHT tools and provide any necessary feedback to Papyrus team.

MDHT Papyrus Support

The MDHT projects release of Papyrus is in response to demand for the CDA tooling in a 100% open source solution.  As such, the MDHT project has decided to release our internal Papyrus integration build to allow early adopters and brave souls the opportunity to use the MDHT tools within the Papyrus UML environment.  The MDHT Papyrus is provided as is and while we will respond to all incidents and issues resulting from using MDHT CDA tools within the environment we can not provide continual updates to the MDHT Papyrus Integration build in response to any Papyrus UML application issues or updates.  We will though provide as part of our development activities release updates as necessary to the MDHT tools in response to issue within the MDHT tool set.

Exploring the UML Class Diagrams in Papyrus

The latest version of Papyrus has limited UML Diagram support but does support the UML Class Diagram. UML Class Diagrams are static representation of the UML models class, attributes, and associations. The Papyrus Example Model includes two UML Class Diagrams, My Document diagram and My Section diagram. For more information on UML Class Diagrams, please refer to the following UML Class Tutorial written by Robert C. Martin.

My Document UML Class Diagram

The My Document UML Class Diagram depicts four classes, Continuity of Care Document (from the CCD Model), My Document, Section, and My Section (from the CDA Model).

The two closed unfilled arrows starting from My Document and My Section and pointing to Clinical Care Document and Section are examples of UML generalization used extensively with the CDA approach. The resulting CDA semantics are that My Document will inherit all constraints as defined in the Clinical Care Document Implementation Guide. The same holds true for My Section which will inherit all constraints define in the CDA standard as it pertains to Section.

The open arrow starting from My Document and pointing to My Section is an example of an UML association. Associations are used to define navigation and cardinality amongst UML classes. In this case, the association defines a relationship between My Document and My Section where My Document will contain 1 to n instances of My Section.

(click for full size)

My Section UML Class Diagram

You can switch to My Section UML Class Diagram by selecting My Section Tab at or near the bottom of your diagram. The My Section UML Class Diagram also depicts for classes, Section (from the CDA Model), My Section, Problem List (from the CDA Model), and My Activity. Once again we have two examples of inheritance and one association.

Looking closer at the My Activity class, you will notice a series of attributes define (or redefined to be exact) in order to further constrain the CDA Problem List definition to support a specific business event or activity. The attributes class code, mood code, code and status are now designated as required for any instance of My Activitiy instead of the optional as they are defined in the Problem List.

Additional CDA Constraints

The CDA approach not only supports cardinality constraints as discussed in the previous section but also allows further constraints of the actual values within the document. The MDHT CDA approach supports these additional constraints by providing a properties tab within which you can supply additional constraints to your CDA document.

We will explore how to apply and view the additional constraints in the next section which will discuss the CDA Tools tab in the Properties view.

CDA Tools tab in the Properties view

The CDA Tools tab in the Properties view is not unique to the Papyrus Platform, in fact it is the same component for all the support MDHT platforms. The current functionality supports a simplified approach to defining CDA constraints for template ids, associations, and codes. Additionally, you are able to customize constraint severity levels and messages used while validating CDA document instances. We will start off looking at validation error levels and messages.

The CDA Tools tab is part of the Eclipse Properties View and is accessible by selecting the CDA Tab shown in the left side of the Properties View.

(click for full size)

CDA Severity Level Constraints and Messages

The MDHT CDA Tools provides a simple approach to defining and customizing the CDA models constraints Severity and Message. For Example, select within/highlight the My Activity class in the My Section UML Class and notice the MDHT CDA Tools Content.

CDA Tools tab in the Properties view for Class Level Constraints

(click for full size)

Validation severity (a property of the <<associationValidation>> stereotype) is used to differentiate whether the relationship is strictly required for an instance to be considered valid. Validation severity may take on one of three possible values ERROR (strictly required, SHALL), WARNING (recommended/best practice, SHOULD), and INFO (optional, MAY). You can use the CDA Tools tab in the Properties view for a Class to define the following constraints

  • CDA Template ID
  • CDA Template Assigning Authority
  • CDA Validation Severity
  • CDA Message

The type of additional constraints are context sensitive. For example. select the code attribute defined in MyActivity and notice that values and constraint types are different.

CDA Tools tab in the Properties view for Code Attribute Constraints

(click for full size)

You can use the CDA Tools tab in the Properties view for a Coded Element to define the following constraints

  • Vocabulary Constraints
    • Code System ID
    • Code System Name
    • Version Date
    • Code
    • Code Display Name
  • CDA Constraints
    • Validation Severity
    • Message

The last type of additional constraints are for UML associations.

These CDA constraints support conformance rules such as: MyDocument SHALL contain exactly one MySection and MySection SHOULD contain one or more MyActivity Additional information about the conformance rule can be specified in property values for the <<associationValidation>>, <<entry>>, and <<entryRelationship>> stereotypes.
